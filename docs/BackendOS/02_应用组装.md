## 4. 关键设计点

| 设计点 | 实现方式 | 为什么 |
|--------|----------|--------|
| **组装顺序严格固定** | 6步按固定顺序执行：异常处理→CORS→中间件→预热→路由→元数据 | FastAPI 中间件执行顺序与注册顺序相反（后注册先执行）。路由匹配按注册顺序，先注册先匹配。顺序错误会导致 CORS 头缺失、异常格式不统一等问题 |
| **CORS 三级降级** | `resolve_allowed_origins()`：环境变量 → 配置文件 → 默认值（L119） | 生产环境通过环境变量注入（无需改代码），开发环境用默认值（零配置），配置文件作为中间方案。灵活性递减，安全性递增 |
| **`load_server_config_or_none()` 容错** | 配置加载失败返回 None 而非抛异常（L85-91） | 配置文件缺失不应阻止应用启动。缺少配置时各组件使用内置默认值，保证服务可用性 |
| **预热依赖** | `warm_application_dependencies()` 提前初始化（L141-158） | 首次请求时才初始化会导致冷启动延迟（LLM 配置加载、容器构建等耗时操作）。预热确保第一个请求就能快速响应 |
| **预热失败不影响启动** | 每个预热步骤单独 try/except（L148-158） | 依赖初始化失败不应导致整个服务不可用。降级运行比完全不可用更好 |
| **禁用默认文档路由** | `docs_url=None, redoc_url=None`（L250-251） | 使用自定义的 RapiDoc（`/rapidoc`）替代默认 Swagger UI，提供更好的 API 文档体验 |
| **`lifespan` 绑定** | `FastAPI(lifespan=app_lifespan)`（L253） | 将启动校验逻辑与 FastAPI 生命周期绑定，确保在 uvicorn 完成加载后、开始接收请求前执行校验 |

### 深入理解：组装顺序为什么重要

```
中间件执行顺序（请求进入方向）：

  Request
    │
    ▼
  ┌──────────────┐   ← 最后注册（Step 4 中第3个 add_middleware）
  │   Timeout    │      最外层，最先执行
  └──────┬───────┘
         ▼
  ┌──────────────┐   ← 第2个注册
  │  RateLimit   │
  └──────┬───────┘
         ▼
  ┌──────────────┐   ← 最先注册（Step 4 中第1个 add_middleware）
  │ RequestLog   │      最内层，最后执行
  └──────┬───────┘
         ▼
  ┌──────────────┐
  │ Route Handler│
  └──────────────┘

为什么 Timeout 在最外层？
→ 超时控制应覆盖所有中间件，包括限流和日志的执行时间

为什么 CORS 在中间件之前配置？
→ CORS 是 Starlette 级别的中间件，需要在 FastAPI 中间件之前处理 OPTIONS 预检请求

为什么异常处理器最先注册？
→ 确保后续步骤（中间件、路由）抛出的异常都能被统一捕获和格式化
```

### 深入理解：CORS 三级降级

```python
# L119: return env_cors or config_cors or list(DEFAULT_CORS_ORIGINS)
#
# 三级降级逻辑：
#
# 优先级 1：环境变量 CORS_ORIGINS=https://app.example.com,https://admin.example.com
#   → 生产部署时通过 K8s/Docker 环境变量注入，无需修改代码或配置文件
#
# 优先级 2：配置文件 server_config.yaml 中的 cors_origins 字段
#   → 开发/测试环境使用，方便版本管理
#
# 优先级 3：DEFAULT_CORS_ORIGINS = ("http://localhost:33001", "http://localhost:38000")
#   → 本地开发的零配置默认值
#
# Python 的 or 链式求值：第一个非空值生效
# env_cors=[] → falsy → 尝试 config_cors
# config_cors=[] → falsy → 使用 DEFAULT_CORS_ORIGINS
```
