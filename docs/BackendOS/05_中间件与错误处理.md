```
┌──────────────────────────────────────────────────────────────────────┐
│              中间件执行顺序（请求进入方向）                            │
└──────────────────────────────────────────────────────────────────────┘

  HTTP Request
       │
       ▼
  ┌─────────────────────────────────────────┐
  │  TimeoutMiddleware                      │  L257  ← 最外层
  │  dispatch():                            │  L271
  │    asyncio.wait_for(call_next, timeout) │
  │    超时 → 504 + record_http_timeout()   │
  └────────────────┬────────────────────────┘
                   │ 未超时
                   ▼
  ┌─────────────────────────────────────────┐
  │  RateLimitMiddleware                    │  L137  ← 中间层
  │  dispatch():                            │  L206
  │    _check_rate_limit(key)               │
  │    超限 → 429 + record_rate_limit()     │
  └────────────────┬────────────────────────┘
                   │ 未超限
                   ▼
  ┌─────────────────────────────────────────┐
  │  RequestLoggingMiddleware               │  L69   ← 最内层
  │  dispatch():                            │  L84
  │    1. 生成/透传 request_id, trace_id    │
  │    2. bind_request_context()            │
  │    3. RequestMetricsTimer() 开始计时    │
  │    4. call_next(request)                │
  │    5. 记录指标 + 结构化日志             │
  │    6. 写入响应头 X-Process-Time 等      │
  │    7. reset_request_context()           │
  └────────────────┬────────────────────────┘
                   │
                   ▼
  ┌─────────────────────────────────────────┐
  │  Route Handler (业务路由)               │
  │  如: /api/chat/stream                   │
  └─────────────────────────────────────────┘

  注册顺序（setup_middleware, L303-330）：
  app.add_middleware(RequestLoggingMiddleware)  ← 最先注册 = 最内层
  app.add_middleware(RateLimitMiddleware, ...)   ← 第二注册 = 中间层
  app.add_middleware(TimeoutMiddleware, ...)     ← 最后注册 = 最外层

  关键规则：FastAPI 中间件"后注册先执行"（洋葱模型）
```

### 滑动窗口限流算法详解

```
┌──────────────────────────────────────────────────────────────────────┐
│              滑动窗口限流算法                                         │
│              max_requests=5, window=60s                              │
└──────────────────────────────────────────────────────────────────────┘

  时间轴（秒）:  0    10   20   30   40   50   60   70   80
                 │    │    │    │    │    │    │    │    │
  请求时间戳:    ●    ●    ●    ●    ●              ●    ●
                 t=0  t=10 t=20 t=30 t=40           t=70 t=80

  ── 第 70 秒的请求到达 ──

  Step 1: _clean_old_requests(key, current_time=70)
  ┌─────────────────────────────────────────┐
  │  window_start = 70 - 60 = 10            │
  │  移除 t=0（< 10，已滑出窗口）           │
  │  保留 t=10, t=20, t=30, t=40            │
  └─────────────────────────────────────────┘

  Step 2: _check_rate_limit(key)
  ┌─────────────────────────────────────────┐
  │  current_count = 4 (t=10,20,30,40)      │
  │  remaining = 5 - 4 = 1                  │
  │  current_count < max_requests → 放行    │
  │  记录 t=70 到队列                       │
  └─────────────────────────────────────────┘

  ── 第 80 秒的请求到达 ──

  Step 1: _clean_old_requests(key, current_time=80)
  ┌─────────────────────────────────────────┐
  │  window_start = 80 - 60 = 20            │
  │  移除 t=10, t=20（< 20，已滑出窗口）   │
  │  保留 t=30, t=40, t=70                  │
  └─────────────────────────────────────────┘

  Step 2: _check_rate_limit(key)
  ┌─────────────────────────────────────────┐
  │  current_count = 3 (t=30,40,70)         │
  │  remaining = 5 - 3 = 2                  │
  │  current_count < max_requests → 放行    │
  │  记录 t=80 到队列                       │
  └─────────────────────────────────────────┘

  对比固定窗口：
  ┌─────────────────────────────────────────┐
  │  固定窗口在窗口边界处有突发问题：       │
  │  第 59 秒发 5 个 + 第 61 秒发 5 个     │
  │  = 2 秒内 10 个请求，但两个窗口都"合规"│
  │                                         │
  │  滑动窗口无此问题：                     │
  │  任何 60 秒区间内都严格 ≤ 5 个请求      │
  └─────────────────────────────────────────┘
```

## 4. 关键设计点

| 设计点 | 实现方式 | 为什么 |
|--------|----------|--------|
| **中间件注册顺序（外→内）** | `setup_middleware()` 先注册 RequestLogging（最内层），最后注册 Timeout（最外层）（L323-330） | FastAPI 中间件"后注册先执行"。超时控制应覆盖所有中间件（包括限流和日志的执行时间），所以 Timeout 必须在最外层 |
| **滑动窗口而非固定窗口** | `_check_rate_limit()` 记录每个请求的精确时间戳，只统计窗口内的请求数（L177-193） | 固定窗口在边界处有突发问题（第59秒和第61秒各发满额请求=2秒内双倍流量）。滑动窗口任何时刻都严格限制窗口内请求数 |
| **ContextVar 传递请求上下文** | `bind_request_context()` 将 request_id/trace_id 绑定到 ContextVar（L96-102） | 异步环境中，全局变量会被不同协程共享导致数据泄漏。ContextVar 是协程安全的，每个协程有独立的上下文副本 |
| **结构化日志** | `emit_structured_log()` 输出 JSON 格式，自动注入 request_id 和 trace_id（L122-143） | 纯文本日志难以机器解析。JSON 格式便于 ELK/Loki 等日志平台检索和聚合，request_id 实现全链路追踪 |
| **健康检查路径排除限流** | `_default_exclude_paths()` 排除 `/api/health`、`/api/ready`、`/api/live` 等（L44-66） | Kubernetes 探针高频调用健康检查，如果被限流会误判服务不可用，触发不必要的重启 |
| **快速失败机制** | `maybe_fail_fast_on_startup()` 在 `fail_fast_startup_validation=True` 时阻止启动（L191-201） | 生产环境中，以不完整状态上线比完全不可用更危险（如 LLM 配置缺失导致所有聊天请求失败）。快速失败让问题在部署阶段暴露 |
| **异常处理器统一格式** | `register_exception_handlers()` 将所有异常转为标准 JSON 格式（L40-60） | 前端只需处理一种错误格式，不需要针对不同异常类型写不同的解析逻辑 |
| **RequestMetricsTimer 跟踪进行中请求** | `__init__` 时 `inc()`，`stop()` 时 `dec()`（L183-199） | 进行中请求数是实时指标，用于监控并发量和发现请求堆积。Counter 无法递减，必须用 Gauge |

### 深入理解：BaseHTTPMiddleware 的 dispatch 模式

```python
# BaseHTTPMiddleware 的工作原理：
#
# class MyMiddleware(BaseHTTPMiddleware):
#     async def dispatch(self, request, call_next):
#         # 请求进入时执行（前处理）
#         ...
#         response = await call_next(request)  # ← 调用下一个中间件/路由
#         # 响应返回时执行（后处理）
#         ...
#         return response
#
# call_next(request) 的含义：
#   - 将请求传递给下一个中间件（或最终的路由处理器）
#   - 等待下游处理完成，返回 Response 对象
#   - 类似于 Express.js 中的 next() 或 Koa 中的 await next()
#
# 洋葱模型：
#   Request → Middleware1.pre → Middleware2.pre → Handler →
#   Middleware2.post → Middleware1.post → Response
```

### 深入理解：ContextVar 为什么安全

```python
# 传统全局变量的问题：
_request_id = "req-001"  # 协程 A 设置
# 协程 B 可能看到 "req-001"，因为全局变量被共享

# ContextVar 的解决方案：
_request_id_var: ContextVar[str] = ContextVar("request_id", default="")

# 协程 A：
token_a = _request_id_var.set("req-001")  # 只影响协程 A 的上下文
# 协程 B：
token_b = _request_id_var.set("req-002")  # 只影响协程 B 的上下文

# 协程 A 中读取：
_request_id_var.get()  # → "req-001"  ← 不受协程 B 影响

# 请求结束后重置：
_request_id_var.reset(token_a)  # 恢复到 set 之前的状态
# 防止协程复用时上下文泄漏
```
