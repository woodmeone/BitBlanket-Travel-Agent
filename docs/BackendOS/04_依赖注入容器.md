## 4. 关键设计点

| 设计点 | 实现方式 | 为什么 |
|--------|----------|--------|
| **register/resolve 分离** | 注册时只存 `(provider, singleton)` 元组，resolve 时才调用 provider | 延迟创建：应用启动时只注册"配方"，不实际创建服务。首次 resolve 时才创建，加快启动速度 |
| **单例 vs 多例** | `singleton=True` 时缓存实例，`False` 时每次创建新实例 | 无状态服务（如 TravelAgent）需要多例（每次对话独立），有状态服务（如 ChatService）需要单例（共享连接池） |
| **签名检测实现热更新** | `_current_container_signature()` 将数据库配置打包为元组，与上次对比（L102-124） | 运维修改数据库配置后无需重启服务。签名变更→重建容器→所有服务使用新配置。实现"配置即代码"的运行时响应 |
| **策略模式选择仓库** | `provide_session_repository()` 根据 `db_backend` 选择 Postgres 或 File 实现（L128-141） | 同一套业务代码支持不同存储后端。开发用 File（零依赖），生产用 Postgres（高可靠）。切换只需改配置 |
| **双重签名检测** | container 层 `_container_signature` + services 层 `_persistence_signature` | 容器层检测到变更会重建整个容器，services 层检测到变更会重置单例。两层防护确保配置变更后所有服务都使用新配置 |
| **providers.py 重新导出** | 从 `bootstrap_services` 重新导出所有 provide_xxx（L27-37） | 解耦：路由代码通过 `dependencies.providers` 导入，不直接依赖 `bootstrap_services` 的内部实现。如果 bootstrap_services 重构，只需修改 providers.py |
| **TravelAgent 非单例** | `provide_travel_agent()` 每次创建新实例（L279-300） | Agent 是有状态的对话图，每次对话需要独立的 Agent 实例。如果复用，不同用户的对话状态会互相干扰 |

### 深入理解：IoC 容器 vs 直接 new 对象

```
  传统方式（紧耦合）：
  ┌─────────────────────────────────────────┐
  │  class ChatRoute:                       │
  │    def handle():                        │
  │      repo = FileSessionRepository()     │  ← 硬编码实现
  │      service = ChatService(repo)        │  ← 手动组装依赖
  │      return service.chat()              │
  │                                         │
  │  问题：                                 │
  │  1. 切换到 Postgres 需要改所有 new 的地方│
  │  2. 测试时无法替换为 Mock               │
  │  3. 依赖关系分散在各处，难以追踪         │
  └─────────────────────────────────────────┘

  IoC 容器方式（松耦合）：
  ┌─────────────────────────────────────────┐
  │  # 注册（只在一处）                      │
  │  container.register("SessionRepository", │
  │                     provide_session_repo)│
  │  container.register("ChatService",       │
  │                     provide_chat_service)│
  │                                         │
  │  # 使用（任何地方）                      │
  │  class ChatRoute:                       │
  │    def handle():                        │
  │      service = container.resolve(        │
  │        "ChatService")                   │  ← 只依赖名称
  │      return service.chat()              │
  │                                         │
  │  优势：                                 │
  │  1. 切换实现只需改 provide_xxx           │
  │  2. 测试时 register Mock 即可            │
  │  3. 依赖关系集中在 register 处           │
  └─────────────────────────────────────────┘
```

### 深入理解：策略模式选择仓库

```python
# provide_session_repository() 的策略选择（L115-142）
#
# 配置文件中：
#   database:
#     backend: postgres          # 或 file（默认）
#     postgres_dsn: postgresql://...
#
# 运行时根据 backend 值选择不同实现：
#
#   backend="postgres" → PostgresSessionRepository
#     - 数据存储在 PostgreSQL
#     - 支持连接池、事务、并发
#     - 适合生产环境
#
#   backend="file"（默认）→ FileSessionRepository
#     - 数据存储在本地 JSON 文件
#     - 零依赖，开箱即用
#     - 适合开发/测试环境
#
# 切换方式：只需修改配置文件，无需改代码
# 这就是策略模式（Strategy Pattern）的核心思想
```
