# 06 路由层与服务解析

> **一句话定位**：路由层是 HTTP 请求进入后端的第一站，通过 service_resolver 从 IoC 容器获取服务实例，再由各路由处理器调用服务方法完成业务逻辑。

## 3. 核心流程总结

### 3.1 请求处理通用流程

```
HTTP Request
    │
    ▼
┌──────────────────────┐
│   FastAPI Router      │  URL 匹配 + 参数校验
│   (chat/session/...)  │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  service_resolver     │  get_xxx_service()
│  .resolve_service()   │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│  Container.resolve()  │  IoC 容器按名称查找
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│   Service 实例        │  业务逻辑执行
│  (ChatService/...)    │
└──────────┬───────────┘
           │
           ▼
┌──────────────────────┐
│   HTTP Response       │  JSON / SSE 流
└──────────────────────┘
```

### 3.2 stream_chat 核心流程

```
POST /chat/stream
    │
    ▼
┌──────────────────────────────────────┐
│  stream_chat()                       │
│  ├─ get_chat_service()               │  从 IoC 获取服务
│  ├─ 提取 request_id / trace_id       │  追踪信息
│  └─ StreamingResponse(               │
│       service.stream_chat(...)       │  委托给 ChatService
│       media_type="text/event-stream" │
│       headers:                       │
│         Cache-Control: no-cache      │
│         Connection: keep-alive       │
│         X-Accel-Buffering: no        │  ← 禁止 Nginx 缓冲
│         X-Request-ID / X-Trace-ID    │
│     )                                │
└──────────────────────────────────────┘
```

### 3.3 错误处理流程

```
业务异常 / 校验失败
    │
    ▼
┌────────────────────────────┐
│  raise_api_error()          │
│  ├─ error_detail()          │  构建 {"success":False, "error":"...", "code":"..."}
│  └─ raise HTTPException     │
│      status_code + detail   │
└────────────────────────────┘

请求校验失败 (Pydantic)
    │
    ▼
┌────────────────────────────┐
│  validation_error_detail()  │
│  ├─ 遍历 exc.errors()       │
│  ├─ 提取 field/message/    │
│  │   issueType/input       │
│  └─ error_detail()          │  统一格式
└────────────────────────────┘
```

## 4. 关键设计点

| 设计点 | 实现 | 为什么 |
|--------|------|--------|
| IoC 容器解耦 | `service_resolver` → `container.resolve(name)` | 路由层不直接 import 服务实例，方便替换实现和单元测试 mock |
| 统一错误格式 | `error_detail()` + `raise_api_error()` | 前端可统一根据 `code` 字段处理错误逻辑，如显示不同提示 |
| SSE 响应头 | `X-Accel-Buffering: no` + `Cache-Control: no-cache` | Nginx 默认会缓冲代理响应，导致 SSE 事件延迟推送；禁止缓冲确保实时性 |
| K8s 探针分离 | `readiness_check` vs `liveness_check` | 就绪探针判断"能否接收流量"（模型加载完了吗？），存活探针判断"进程还活着吗？"，职责不同 |
| readiness 快照 | `app.state.readiness_snapshot` | 启动时一次性校验并保存快照，运行时直接读取，避免每次探针都做重量级检查 |
| 路径参数校验 | `Annotated[str, Path(pattern=...)]` | 利用 FastAPI 的 Annotated 类型注解，声明式校验 + 自动生成 OpenAPI 文档 |
| 模型合法性校验 | `set_session_model` 中校验 `known_model_ids` | 防止用户绑定不存在的模型，导致后续聊天请求失败 |
| 校验错误转换 | `validation_error_detail()` | FastAPI 原始校验错误格式复杂（嵌套 loc 元组），转换为前端友好的 `field.message` 格式 |

## 5. 你要回答的问题

1. **为什么路由层不直接 `from services import ChatService` 创建实例，而是通过 `service_resolver` + IoC 容器？** 如果直接创建实例会有什么问题？

2. **`stream_chat` 为什么要设置 `X-Accel-Buffering: no` 响应头？** 如果不设置，在 Nginx 反向代理后面会发生什么？

3. **K8s 的 readiness 和 liveness 探针有什么区别？** 如果把 readiness 的逻辑放到 liveness 里，会有什么风险？

4. **`raise_api_error` 和直接 `raise HTTPException` 有什么区别？** 为什么项目要封装一层？

5. **`set_session_model` 为什么要先校验模型 ID 是否在可用列表中？** 如果不校验，后续流程会在哪里失败？

6. **`service_resolver` 的每个 `get_xxx_service()` 函数内部都调用 `resolve_service(name)`，为什么不用类型参数（如泛型）而是用字符串名称？** 字符串名称有什么优缺点？
