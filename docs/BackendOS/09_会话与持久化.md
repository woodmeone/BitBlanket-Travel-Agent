> **一句话定位**：会话与持久化层采用门面+仓储模式，SessionService 作为门面委托 SessionLifecycleService 处理 CRUD，SessionRepository 抽象接口支持 File/Postgres 两种实现，File 实现通过原子写入+备份恢复保证数据可靠性。

## 3. 核心流程总结

### 3.1 会话服务调用链

```
Route Handler (session.py)
    │
    ▼
SessionService (门面)
    │  委托
    ▼
SessionLifecycleService (生命周期编排)
    │
    ├─ CRUD 操作 ─► SessionRepository (抽象接口)
    │                    │
    │                    ├─ FileSessionRepository (JSON 文件)
    │                    │    ├─ 内存 dict 缓存
    │                    │    ├─ _atomic_write_json() 持久化
    │                    │    └─ .bak 备份恢复
    │                    │
    │                    └─ PostgresSessionRepository (PostgreSQL)
    │                         ├─ SQLAlchemy Engine
    │                         ├─ sessions_table + session_messages_table
    │                         └─ asyncio.to_thread 同步→异步桥接
    │
    └─ 记忆副作用 ─► SessionMemoryManager (Protocol)
                         ├─ delete_session()
                         └─ clear_session_messages()
```

### 3.2 原子写入流程

```
_atomic_write_json(path, payload)
    │
    ├─ 1. 创建临时文件
    │      tempfile.mkstemp(prefix=".sessions.json.", suffix=".tmp")
    │
    ├─ 2. 写入 JSON + flush + fsync
    │      json.dump(payload, tmp_file)
    │      tmp_file.flush()
    │      os.fsync(tmp_file.fileno())     ← 确保数据落盘
    │
    ├─ 3. os.replace(temp_path, path)      ← 【核心】原子性替换
    │      要么完全成功（新文件），要么完全不变（旧文件）
    │
    ├─ 4. _fsync_directory(target_dir)     ← 确保目录项更新持久化
    │
    └─ 5. [清理] 如果临时文件残留则删除
```

### 3.3 备份恢复流程

```
_load_from_file()
    │
    ├─ 尝试加载主文件 (_load_json_file)
    │      │
    │      ├─ 成功 → 返回数据
    │      │
    │      └─ 失败（文件不存在/JSON损坏）
    │            │
    │            ▼
    │         尝试加载备份文件 (.bak)
    │            │
    │            ├─ 成功 → 用备份数据恢复主文件 → 返回数据
    │            │
    │            └─ 失败 → 返回空字典 {}（全新状态）
```

### 3.4 PostgreSQL 仓储消息存储

```
create(session_data)
    │
    ├─ 构建 session 行 → INSERT/UPDATE sessions_table
    │
    └─ 构建 message 行 → _replace_messages_sync()
           │
           ├─ DELETE session_messages WHERE session_id = ?
           │
           └─ INSERT session_messages (session_id, sequence, role, content, ...)

get(session_id)
    │
    ├─ SELECT sessions_table WHERE session_id = ?
    │
    └─ SELECT session_messages_table WHERE session_id = ? ORDER BY sequence
           │
           ├─ 有消息行 → 返回规范化消息列表
           │
           └─ 无消息行 → 回退到 sessions.messages JSON shadow
```

## 4. 关键设计点

| 设计点 | 实现 | 为什么 |
|--------|------|--------|
| 门面模式 | `SessionService` 委托 `SessionLifecycleService` | 对外接口稳定，内部可自由重构；门面可添加横切逻辑（日志、缓存）而不影响协作者 |
| 仓储模式 | `SessionRepository(ABC)` + 两种实现 | 抽象接口解耦业务逻辑和存储实现，开发环境用 File，生产环境用 Postgres，切换无需改业务代码 |
| 原子写入 | 临时文件 → fsync → os.replace → fsync 目录 | 防止写入中断导致数据损坏：进程崩溃时目标文件要么是旧版本（完整），要么是新版本（完整） |
| 备份恢复 | 主文件损坏时自动从 .bak 恢复 | 增加数据可靠性层：即使主文件损坏，仍可从备份恢复，降低数据丢失风险 |
| asyncio.Lock | File 仓库中所有写操作加锁 | 防止并发写入导致数据竞争：同一时刻只有一个协程在修改会话数据 |
| asyncio.to_thread | 文件 I/O 和数据库操作放到线程池 | 避免阻塞事件循环影响其他请求的响应速度 |
| 延迟记忆绑定 | `memory_manager_factory` 延迟创建 | Agent 模块导入较重（LangChain、模型加载），延迟到首次使用时才创建，轻量操作不受影响 |
| Protocol 合约 | `SessionMemoryManager(Protocol)` | 鸭子类型，实现类无需显式继承，测试替身更容易创建 |
| JSON 类型适配 | `JSON().with_variant(JSONB, "postgresql")` | PostgreSQL 的 JSONB 支持索引和高效查询，SQLite/MySQL 使用通用 JSON 类型 |
| 消息表分离 | `session_messages_table` 独立于 `sessions_table` | 消息是高频读写的大字段，分离后查询会话列表不需要加载全部消息，提升性能 |
| 级联删除 | `ForeignKey("sessions.session_id", ondelete="CASCADE")` | 删除会话时自动删除关联消息，避免孤儿数据 |
| JSON shadow | Postgres 实现中 sessions.messages 保留空数组 | 兼容性过渡：消息真实数据在 session_messages 表，sessions.messages 为空数组，旧代码仍可读取 |
